## 2. 架构技术债

### 🟡 TB-ARCH-01: 单进程无集群模式

**现状**：`server.js` 单进程运行，未使用 Node.js `cluster` 或 PM2 多实例。

**影响**：
- 单核 CPU 利用率，无法充分利用多核
- 进程崩溃后无自动重启
- 不适用生产环境高可用要求

**建议**：添加 `pm2` 配置或 `cluster` 模式启动脚本。

---

### 🟡 TB-ARCH-02: state.json 到 SQLite 的迁移路径未清理

**现状**：`db.js` 已实现 SQLite 5 张表，`state.js` 内部已切换到 `db.js`，但 `lib/state.js` 中仍保留从 `state.json` 自动迁移的逻辑（`migrateFromJson`）。

**影响**：
- 迁移代码为一次性逻辑，长期保留增加认知负担
- `state.json` 文件可能残留在磁盘，误以为仍在使用

**建议**：将 `migrateFromJson` 提取到独立脚本 `scripts/migrate-state.js`，`state.js` 启动时仅检测并提示运行迁移命令。

---

### 🟡 TB-ARCH-03: feishu-api.js 的实现与实际启用脱节

**现状**：`lib/context/feishu-api.js` 已实现完整的飞书 Open API 调用逻辑（token 换发、日历拉取、缓存刷新），但 `feishu.js` 中仅在 `process.env.FEISHU_APP_ID` 存在时才调用它。当前项目 `.env.example` 未包含此变量，`docs/feishu-integration.md` 仍标记为"草案"。

**影响**：代码路径存在但 99% 用户无法激活，形成死代码

**建议**：在 `.env.example` 中补充 `FEISHU_APP_ID` / `FEISHU_APP_SECRET` 注释，将 `feishu-integration.md` 标记从"草案"更新为"已实现（需配置）"。

---

### 🟢 TB-ARCH-04: 零 TypeScript / 零构建管线

**现状**：纯 JavaScript ESM，无编译、无类型检查、无打包。

**影响**：长期维护中类型错误只能在运行时暴露，重构风险较高。当前代码量 (~3000 行 JS) 尚可接受。

**建议**：在代码量超过 5000 行前引入 JSDoc 类型注释 + `tsc --checkJs` 过渡方案。

## 3. 测试覆盖技术债

### 🟡 TB-TEST-01: 测试覆盖率极低

**现状**：`npm test` 仅覆盖 `lib/llm/_parse.js` 和 `lib/router.js` 两个文件（37 项测试）。

**未覆盖的关键路径**：

| 模块 | 风险 | 建议测试 |
|------|------|---------|
| `lib/state.js` | 核心数据持久化，CRUD 错误会丢历史 | 单元：append/prune/clear |
| `lib/llm/index.js` | LLM provider 切换逻辑 | 单元：provider 路由 + fallback |
| `lib/llm/fallback.js` | 降级链的正确性 | 单元：各 provider 失败场景 |
| `lib/music/ncm.js` | 直链解析 + 过期检测 | 集成：搜索/播放/refresh |
| `lib/scheduler.js` | 定时任务注册与触发 | 单元：cron 解析 + onTrigger |
| `lib/prompt-builder.js` | 7 时段匹配 + 心情感知 | 单元：各时段输入→输出 |

**建议**：在第 3 周优先补 `state.js`、`ncm.js`、`scheduler.js` 三个模块的测试。

---

### 🟡 TB-TEST-02: 无集成测试 / 端到端测试

**建议**：至少添加 `test/integration/chat.test.js`，使用 `supertest` mock LLM 调用验证 API 契约。

### 🟢 TB-TEST-03: 无 CI 配置

**建议**：添加 GitHub Actions 最小流水线：`npm install → npm test`。

**现状**：`groupGenres()` 内嵌 10 条 `RULES` 数组，每条含 `keywords` 数组，可读性一般但逻辑正确。

### 🟢 TB-CODE-03: feishu-api.js 与 feishu.js 存在功能重叠

**现状**：`feishu.js` 本身包含本地 JSON 解析和 `toPromptFragment`，`feishu-api.js` 提供 Open API 调用。两者通过环境变量切换，但 `feishu.js` 仍保留降级逻辑（API 失败→读本地 JSON）。

**影响**：双数据源逻辑分散在两个文件，调试时需跨文件追踪。

**建议**：合并为 `feishu.js` + `feishu-local.js` + `feishu-api.js` 三层，或使用策略模式统一接口。

## 6. 安全技术债

### 🟡 TB-SEC-01: API_TOKEN 默认不启用

**现状**：`auth.js` 中间件在 `API_TOKEN` 未设时**完全跳过鉴权**，所有 API 端点（含 `POST /api/state/clear`、`POST /api/profile/save` 等破坏性操作）无需任何认证。

**影响**：如果服务绑定到 `0.0.0.0`（非 `127.0.0.1`），局域网内任何设备可清空对话历史、篡改品味文件。

**建议**：
- `.env.example` 中加大警告注释
- 或在启动时检测 `PORT` 绑定地址，非 loopback 且无 token 时打印明显警告

---

### 🟡 TB-SEC-02: WebSocket 无频率限制

**现状**：`/stream` 端点接受连接后无消息频率限制，恶意客户端可高频发送 `ping` 或伪造消息消耗服务端资源。

**建议**：在 `ws.on('message')` 中添加 IP 级别速率限制（如每分钟 30 条非 ping 消息）。
